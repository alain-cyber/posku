How it works
Three steps
  1. Configure your ERP — In Settings, set the base URL, auth header, and endpoint paths for SKUs and POs.
  2. Build & push SKUs — Each product variant gets a code from category, product, and up to two attributes. Saving POSTs the SKU to your ERP and mirrors it locally so the PO builder can reference it.
  3. Assemble & push POs — Pick mirrored SKUs, set quantities, and POST the purchase order. CSV export and print stay available as fallbacks.
Source of truth
Your ERP owns the records. Posku keeps a local mirror in localStorage so the PO builder and recent lists work without re-fetching.

Payload reference
What Posku sends to your ERP

POST {baseUrl}{sku path} — body:

{
  "code": "APPA-TSHI-MEDI-BLAC-001",
  "category": "Apparel",
  "product": "T-Shirt",
  "attributes": { "attr1": "Medium", "attr2": "Black" },
  "unit_cost": 12.50,
  "stock_on_hand": 0,
  "supplier": "Acme Co.",
  "notes": ""
}

POST {baseUrl}{po path} — body:

{
  "number": "PO-0001",
  "supplier": "Acme Co.",
  "ship_to": "Warehouse A",
  "expected_by": "2026-06-01",
  "reference": "",
  "tax_percent": 8.5,
  "lines": [
    { "sku_code": "...", "quantity": 10, "unit_cost": 12.50 }
  ],
  "subtotal": 125.00,
  "total": 135.63
}

If your ERP responds with an id, Posku stores it on the local record.
